# ayepi examples

Runnable example apps, each **exactly three TypeScript files** and ramping from a hello
world up to one that uses (nearly) everything:

| # | App | Run | URL | Teaches |
| --- | --- | --- | --- | --- |
| 01 | [hello](01-hello) | `pnpm hello` | http://localhost:3001 | spec → typed server → typed client; generated docs |
| 02 | [notes](02-notes) | `pnpm notes` | http://localhost:3002 | CRUD: path params, query, validation, `status()`, a declared 404 |
| 03 | [chat](03-chat) | `pnpm chat` | http://localhost:3003 | realtime events over WebSocket + broker fanout |
| 04 | [kitchen-sink](04-kitchen-sink) | `pnpm kitchen-sink` | http://localhost:3004 | a param **loader**, cookies, declared errors, file upload, item streaming, live events — auth via **`@ayepi/auth`** (JWT), observability via **`@ayepi/otel`** + **`@ayepi/log`** |
| 05 | [jobs](05-jobs) | `pnpm jobs` | http://localhost:3005 | background job queue with live progress — **`@ayepi/work`** (zero-config in-memory backend) bridged to ws events |
| 06 | [mcp](06-mcp) | `pnpm mcp` | http://localhost:3006 | expose the spec as **MCP** tools (**`@ayepi/mcp`**) with an interactive tool-explorer |
| 07 | [everything](07-everything) | `pnpm everything` | http://localhost:3007 | the grand tour — core + **auth + rate + log + otel + work + updown + codec + mcp** in one dashboard |
| 08 | [plugins](08-plugins) | `pnpm plugins` | http://localhost:3008 | a **plugin host** (**`@ayepi/plugin`**) — install/uninstall plugins (auth → notes → stats) into a running server, with hot remove/re-add |
| 09 | [cache](09-cache) | `pnpm cache` | http://localhost:3009 | response **caching** (**`@ayepi/cache`**) — a per-user report cached by `ttl` + `vary`, with stale-while-revalidate and manual `bust` |
| 10 | [files](10-files) | `pnpm files` | http://localhost:3010 | a presigned-URL **file store** (**`@ayepi/files`**) — `mountFiles` signs `GET`/`PUT` routes; uploads/downloads stream straight to/from disk |

Node only (no Deno/Bun). The client is a single-file **Vue** app; the server is **Node**
via `@ayepi/node`; the shared file holds the **zod schemas + spec**. Apps 04–07 also wire in
the companion packages (each documented in its own `ayepi-<pkg>.md`).

## The three files

Every app is just:

```
NN-name/
  shared.ts        # zod schemas + spec() — the single source of truth
  server.ts        # implement() the handlers + server() + serve on Node
  client.ts        # a single-file Vue app: build a typed sdk from the manifest, render
  manifest.gen.ts  # generated: the zod-free manifest as plain data (see below)
```

`client.ts` imports two things from the spec, both **without zod**:

- the spec **type** — `import type { api } from './shared'` (erased at build time), which
  types every `sdk.call(...)`;
- the spec **manifest** — `import manifest from './manifest.gen'`, the zod-free routing
  metadata as a **plain data module**.

Because the manifest is an ordinary import of plain data and the spec is imported
type-only, a normal bundler **tree-shakes zod away** — no special build step. So **zero
zod ships to the browser** and the client gets its routing manifest at **build time**.

`manifest.gen.ts` is generated by the shared harness ([`_harness.ts`](_harness.ts) — not
one of an app's "three files") each time a server starts: it writes
`JSON.stringify(app.manifest())` to that file. The harness also bundles each `client.ts`
on demand with esbuild (Vue included) and serves it at `/app.js` behind a minimal HTML
shell. Editing a `client.ts` and refreshing re-bundles it; the manifest refreshes whenever
you restart the server. (The files are committed so `pnpm typecheck` works on a fresh
checkout; in your own app you'd `.gitignore` them and run the codegen as a prebuild step.)

> **Don't care about bundle size?** `client()` also accepts the **spec itself** — pass
> `client({ baseUrl, manifest: api })` (the value, not the type). It derives the manifest
> for you, but since the spec carries zod, that pulls zod into the bundle. The
> `manifest.gen.ts` route exists precisely to avoid that; pick whichever fits your app.
>
> **More than one spec?** Pass extra servers to the harness as `manifests: { admin, … }`;
> the client reads them via `import { manifests } from './manifest.gen'` (`manifests.admin`).

## Run one

From the repo root, build the workspace packages once, then start an app:

```sh
pnpm -r build                      # build @ayepi/core, @ayepi/node, … (the examples import their dist)
pnpm --filter @ayepi/examples hello
# or: cd examples && pnpm hello
```

Open the URL from the table. Each server also prints every generated-docs URL on start:

```
  01 · hello
  ────────────────────────────────────────────
  App         http://localhost:3001/
  Swagger UI  http://localhost:3001/docs/swagger
  ReDoc       http://localhost:3001/docs/redoc
  AsyncAPI    http://localhost:3001/docs/asyncapi
  OpenAPI     http://localhost:3001/docs/openapi.json
  AsyncAPI    http://localhost:3001/docs/asyncapi.json
```

## Generated docs (every app)

The spec produces docs with **no extra work** — `server(api, …, { docs: true })` serves:

| URL | What |
| --- | --- |
| `/docs/swagger` | Swagger UI (OpenAPI 3.1) |
| `/docs/redoc` | ReDoc (OpenAPI 3.1) |
| `/docs/openapi.json` | the OpenAPI document |
| `/docs/asyncapi` | AsyncAPI 3.0 viewer (the WebSocket events) |
| `/docs/asyncapi.json` | the AsyncAPI document |

## Test them

- **In the browser** — open the app URL and click around. For **03-chat**, open it in two
  tabs on the same room to watch realtime fanout. For **04-kitchen-sink**, log in (try
  `admin` for the admin role, `blocked` for a typed 403), start a job and watch the progress
  bar update from live events, then "Stream log" to see an item stream arrive line by line.
  For **05-jobs**, enqueue a compute job and watch it run on the background work engine; for
  **06-mcp**, pick a tool and invoke it; for **07-everything**, hammer **Ping** to trip the
  rate limit and open the **Snapshot** panel to see `Date`/`Map`/`Set` decoded by `@ayepi/codec`.
- **With curl** — every endpoint is a normal HTTP route:
  ```sh
  curl -XPOST localhost:3001/greet -H 'content-type: application/json' -d '{"name":"Ada"}'
  curl localhost:3002/listNotes
  TOKEN=$(curl -s -XPOST localhost:3004/login -H 'content-type: application/json' -d '{"user":"demo"}' | jq -r .token)  # a signed JWT
  curl localhost:3004/me -H "authorization: Bearer $TOKEN"
  curl -XPOST localhost:3005/enqueue -H 'content-type: application/json' -d '{"label":"x","n":5}' && curl localhost:3005/listJobs
  curl -XPOST localhost:3006/callTool -H 'content-type: application/json' -d '{"name":"add","args":{"a":2,"b":3}}'
  ```
- **Type-check** the whole examples package: `pnpm --filter @ayepi/examples typecheck`.

Each app's own `README.md` lists its endpoints and what to try.
